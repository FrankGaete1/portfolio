Design Lifecycle & considerations

Typically, my design lifecycle includes the following phases after receiving a project request and processing the initial information provided. This is a high level overview with some detail provided for each phase. Each project has its own unique nuances and challenges, for the sake of brevity the following outline is intentionally general.

Understand Requirements

The design lifecycle here includes the following phases after receiving a project request and processing the initial information provided.

Assuming the requestor is a business stakeholder such as a product manager, I

  • Schedule a design session that includes the business stakeholder, business analyst, dev lead, and other designer(s) as needed
  • Review requirements, business goals and any available UX research
  • Engage the group in a sketching exercise and iterating through various concepts quickly. The main goal here is to confirm I understand the requirements and develop a design direction that is acceptable with all represented areas and establish initial target dates for deliverables.

Patterns & Principles

Once a solid understanding of the requirements and goals has been established, the next step is to review existing UI/UX design priniciples and patterns such as a style guide, design pattern library, best practices and more.

Depending on the project we may be able to use existing layouts, styles or design patterns. There may be cases where we need to create or modify a new style or design pattern.

Ultimately, basing our development on established and enforced guidelines help provide a consistency across applications and corporate communication which adds to a better user experience.

Wireframes & Mockups

After determining the layouts, styles and design patterns we will use for our initial design concepts, we can start putting together wireframes.

Wireframes should be monochramitic to avoid distractions such as "why are using that shade of blue?" and allows the reviewing audience to focus on the important elements for which wireframes are intended. These elements include interface elements we need to display (text, images, buttons, etc), general layout of these elements, relationships between elements and the workflows between elements and/or views.

Once we have approved wireframes, we use them along with our defined styles, design patterns, etc, to create a "high fidelity" design mockup that demonstrates what a view or set of views will look like in production.

Prototypes & Integration

Depending on factors such as project timeline, overall capacity or defined design approach, we may skip one of the previous phases (wireframe or mockup) and go straight to prototyping.

In the last few years, I have used Twitter's Bootstrap framework for my rapid prototyping. Benefits of using Bootstrap include: huge popularity and use so it's well documented, easy to use, ability to manipulate layouts within the 12 column grid system, responsive, cross-browser issues have been worked out, and more.

Prototyping also provides a great time and money savings as we can quickly provide a working HTML/CSS/jQuery sample that allows stakeholders and beta users to get a very good idea of what the views will look and behave like as a user interacting with them. Zero or minimal effort from software engineers is expended until the prototype is approved for the development phase.

 

Project Insights Example: ICS Portal

Here I describe how design considerations and lifecycle were applied to one of the largest projects I worked on, Broadrige's ICS Portal. This project included several "Bank and Broker" applications, was a "first-of-its-kind" at Broadridge and took over 1 year to complete.

Problem

Create a single sign-on portal for several "Bank and Broker" applications which were designed and developed in silos which resulted in disparate designs, workflows, environments and more. A client (user) licensing several of these applications had to sign in to each application individually and would be restricted to viewing data from only 1 application at a time. We needed to offer users true "SSO" (single sign on) and allow them to view and leverage information, data and assets across all licensed applications.

Solution

Create a portal that included several applications with a consistent and modern UI/UX design, single sign-on functionality, security features, dashboard views with cross-application data views without having to redesign/redevelop each standalone application. Sounds easy right?

Planning

There were several initial concurrent efforts including portal software (ie., third party or write our own); select applications to include; review client feedback that supported the need to offer a portal solution; review standalone applications' design challenges; resource planning; cost estimating and other items that fell into the "sprint 0" phase.

UI/UX Approach

As the UI/UX team lead, I was involved in every design related aspect of this project, including "sprint 0" items such as portal design inspirations, UX research and exploring third party portal software to understand what UI/UX features and functionality it offered.

To avoid redesigning/redeveloping entire standalone applications, the software engineers created APIs into parts of each standalone app to pull data and functionality in. This allowed us to design a consistent and moderan UI/UX for all applications in the portal, leveraging some existing styles and design patterns, but in many cases we created new styles and patterns.

This project was enormously challenging but we had a lot of fun with it and received great feedback from our clients.
